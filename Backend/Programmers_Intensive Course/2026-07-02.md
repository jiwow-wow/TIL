[TIL] 2026-07-02 | DDD 4계층 구조와 전략적 설계

1. 4계층
표현: 외부 <-> 내부 언어로 통역(통역사) - Controller
  HTTP 요청, JSON
  핵심 규칙: 도메인 객체를 그대로 API로 노출하지 마라
응용: 트랜잭션 책임(PM) - Service
도메인: 사업의 진짜 규칙(전문가) - Order, Product, Money
인프라: 기술을 실제로 처리(외주 업체, 창고) - Repository

-비교 포인트
빈약한 도메인(x) vs 풍부한 도메인(o)
응용 서비스 = 흐름 vs 도메인 서비스 = 규칙

2. 필수 용어
Entity: 고유 식별자(ID)로 구별되는 객체
VO: 값 객체(ex: 돈)
aggregate: 항상 묶여 다뤄지는 객체 덩어리
불변식: 항상 참이 되어야 하는 규칙(ex: 금액은 음수가 될 수 없다)

3. DIP 의존성 역전
표현 -> 응용 -> 도메인 <-(DIP)- 인프라
도메인 <-(DIP)- 인프라
서비스 <-(DIP)- 인프라

4. 헥사고날
인풋이 많아야 제대로 사용이 가능함
평균적인 웹에서는 트레이드 오프만 느껴질 가능성이 높음

5. 자가진단 안티패턴
폴더는 나눴는데? -> 로직이 한 곳에 뭉쳐있다
빈약한 도메인 -> 엔티티는 게터세터 뿐이고, 진짜 규칙은 서비스의 if문에 있다
레포지토리 인터페이스를 도메인에 둠
엔티티를 API로 직노출 -> 컨트롤러가 엔티티를 그대로 받고 반환

6. 도메인 분류
core: 돈을 버는 이유
supporting: core을 돕지만 차별점
generic: 어느 회사나 똑같이 필요한 범용(ex: 결제)

기타 용어
POJO(Plain Old Java Object): 오래되고 순수한 자바 객체 -> 별로 안 좋은 뜻임
