## Elasticsearch

기존 RDB의 like %키워드%를 활용했으나, 한계가 있음  

RDB 검색이 무너지는 장면  
1. 사용자는 검색창부터 연다  
	-검색은 부가 기능이 아니라 유입의 관문이다  
2. where name Like%키워드%  
	-상품수가 많아지면 
	-오타, 붙여쓰기, 동의어에 대처를 못함  
	-형태소(의미를 가진 가장 작은 말의 단위)
	-정렬기준을 정하는 점수가 없다  
	-한 글자 칠 때마다 풀스캔이 일어남 상품이 많아지면 => 성능 낮아짐  

동의어, 형태소, 오타, 랭킹, 자동완성

ES가 빠른 이유  
1. 검색에만 특화된 DB 옆에 두는 별도의 저장소  
2. 책 뒤의 "찾아보기" 비유 => 역색인  
	데이터를 넣을 때 미리 정리해 둔다  
3. 분석기와 한국어 - nori  
	Character Filter: 전처리  
	Tokenizer: 쪼개기  
	Token Filter: 가공 - 소문자화, 불용어 제거, 동의어 확장 ...  
	-동의어 처리와 자동완성도 이에 해당
검색은 text, 필터는 keyword

데이터 정합성 (ES는 원본이 아니다)
ES 가 포기한 것들  
1. 트랜잭션이 없다  
2. 매핑이 사실상 불변이다  
3. near-realtime:준실시간 -> 작성 직후 1초간은 검색에 안 보일 수 있음  
=> 사본(ES)을 어떻게 원본(DB)과 어긋나지 않게 유지하는가
