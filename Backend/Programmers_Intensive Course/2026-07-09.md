## Docker - 컨테이너박스
(옛날에는 목재, 음식, 자동차 등의 물류를 전부 다른 방식으로 실었으나, 컨테이너라는 규격을 만들어서 담으면 어떤 운송수단이던, 똑같은 방식으로 실을 수 있게 된다)

1. 하는 일
	애플리케이션 + 그 app이 실행되는 데 필요한 모든 것(JDK, 라이브러리, 설정, OS 라이브러리)  
	"실행 환경 자체를 통째로 배송"  
2. 알아야 할 개념  
	1. 이미지: 붕어빵 틀 / 클래스   
	2. 컨테이너: 붕어빵 / 인스턴스  
	3. Dockerfile: 붕어빵 틀 제작 설명서  
3. Docker Compose "컨테이너 10개를 명령어 한 줄로"  
	docker run 10번을 치는 게 아님  
	depends_on: [config-server] 해당 컨테이너는config-server 다음에 실행  
	컨테이너를 하나의 가상 네트워크로 묶어줌  
	서비스 이름이 곧 호스트 이름이 된다  

compose의 한계=> kubernets가 필요해지는 순간  
	1. 서버 한 대로 버틸 수 없게 된다  
	2. 서버 자체가 죽으면 끝남  
	3. 타임세일 기간에만 order-service를 5개로 늘리고 싶다면?  
		자동으로 가감은 못함  
	4. 새 버전을 배포하는 동안 서비스가 끊기지 않게 하려면?  
		무중단 배포를 하기위해  

### 요약:
Docker  = 컨테이너 한 개를 만들고 실행하는 기술  
Docker Compose = 한 대의 서버에서 여러 컨테이너를 다루는 도구  
Kubernetes = 여러 대의 서버에서 수많은 컨테이너를 운영하는 도구  
=> 하지만 결론적을 개발환경에서는 최적이나, 서비스 운영에서 제한사항이 생김 => "kubernets 로!"  

## kubernets(k8s)
명령형 -> 선언형
쿠버네티스는 24시간 쉬지 않고 현재상태와 선언된 상태를 비교
ex) 컨테이너 하나가 죽었다 -> 즉시 하나 더 띄움
=> 자가 치유: 사람이 서버를 돌보는 게 아니라 시스템이 스스로 돌봄

쿠버네티스는 여러 대의 서버를 하나로 묶어서 거대한 컴퓨터 한 대처럼 쓰게 해줌
	클러스터: k8s로 묶인 서버의 전체 집합
	노드: 클러스터를 구성하는 개별 서버
	컨트롤 플레인: 두뇌, 관제탑 => 어떤 컨테이너를 어느 노드에 배치할지 결정	

매니패스트 - 쿠버네티스와 대화하는 방법
=> 원하는 상태를 선언할 때 쓰는 YAML파일

핵심 오브젝트 4형제:  
	1. pod: 컨테이너를 무엇으로 실행하는가   
	쿠버네티스는 컨테이너를 직접 다루지 않고, pod라는 포장지에 한 번 감싸서 다룸, 쿠버네티스에서 최소 배포 단위임  
	파드가 분리되면 서로 다른 IP를 가짐  
	파드의 중요한 2가지 성질  
	-파드는 소모품: 죽으면 그걸로 끝  
	-파드의 IP는 일회용  
	2. deployment: 그 pod를 몇 개 어떻게 유지하는가  
	-파드를 항상 N개로 유지하라  
	3. service: 계속 바뀌는 pod들을 어떻게 안정적으로 찾는가?  
	4. ingress: 외부 요청을 어떻게 들여보내는가  

## ECS
1. 무엇을 실행할 것인가? => task definition  
2. 어떻게 실행할 것이가? => service  
3. 어디에서 실행할 것인가> => cluster  

배치전략(EC2)모드
	1. binpack
	2. spread
	3. random

네트워크 모드: awsvpc
	문제상황
	-옛날방식: Ec2 하나에 여러 컨테이너 -> 포트 충돌
	-보안 그룹을 Ec2단위로만 설정 -> 세밀한 제어 불가
	
	해결책:
	-각 Task가 ENI(독립 네트워크 인터페이스)를 가짐
	-Task마다 private IP 할당
