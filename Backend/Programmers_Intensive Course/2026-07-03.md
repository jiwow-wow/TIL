정산 도메인
정산이란, 받은 돈을 주인에게 제대로

조건
1. 무결성: 1원도 틀리면 안 된다
2. 멱등성: 두 번 하면 안 된다
3. 성능: 재시간에 끝나야 한다

실무에서 정산은 버그가 아니라 사건이 된다 -> 무거운 도메인
즉, 정산은 틀리지 않고 중복되지 않고 재시간에 끝났느냐로 평가

핵심 용어 개념 
주문/결제 (원천 데이터)
-정산의 입력
-원천은 절대 변하지 않아야함(불변)
-취소가 나면 원천을 고치는 게 아니라 취소라는 레코드가 추가
수수료-정률vs정액
-정률은 비율(결제금의 3.3%), 정액은 고정금액(건당 100원)
부가세(VAT)
-보통 수수료에 붙는 10%
정산금(net) 
-판매자가 실제로 받는 돈
-정산금 = 결제금액 - 수수료 - 부가세
정산 주기/마감
-마감 때문에 배치가 정해진 시간 안에 끝나야 하는 압밥(성능)이 생김
정산상태
-Pending(대기) -> DONE(완료)/FAiLD(실패)
정산명세
-왜 이 금액이 입금됐는지 설명한는 내역서

1원의 무결성-왜 돈 계산에 float를 쓰면 안 되나
=> 컴퓨터는 0.1을 정확히 다루지 못함, 부동소수점으로 인해서 
=> double, float는 작은 오차가 쌓임 
=> BigDecimal로 다루자

BigDecimal 주의사항
new BigDecimal(0.1) 처럼 double을 넣으면 오차가 그대로 따라옴 따라서 BigDecimal('0.1") 또는 BigDecimal.valueOf()를 사용

멱등성(Imdempotency)
같은 작업을 여러번 실행해도 결과가 1번 실행한 것과 같음
구조적으로 막는 방법
1. UNIQUE 제약: DB레벨에서 보장하는 최후의 방어선
2. 상태값 확인: 이미 DONE인 건은 건너뛰기 
3. 재시작: spring batch는 job이 어디까지 처리했는지 메타 데이터에 저장, 동일한 job parameter로 재실행하면 실패한 지점부터 이어서 처리함(처음부터 X)

왜 배치인가? -> 대용량 처리의 필요성
findAll()로 한번에 읽어 리스트에 담으면 OOM(OutOfMemory)로 프로세스 다움

-Xmx512m: 이 프로그램은 힙을 최대 512mb까지만 써 라는 명령임

스프링 배치 기초와 chunk지향 처리
-스프링 배치는 Job(전체 작업) -> Step(작업의 한 단계) 구조로 묶임
chunk 지향 처리: 읽고-> 가공하고->쓰기를 정해단 단위(chunk)로 끊어서 반복하자 => 메모리가 떠 있는 데이터가 항상 청크 크기로 일정하게 유지됨 (100만 건이든 1000만건이든 메모리 사용량은 똑같음)
