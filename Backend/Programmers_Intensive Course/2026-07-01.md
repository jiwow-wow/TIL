1. 오늘 배운 핵심 개념
리팩토링 (Refactoring)
정의: 소프트웨어의 겉으로 드러나는 기능은 그대로 둔 채, 내부의 구조를 더 깔끔하고 유연하게 개선하는 행위.

목적: 가독성을 높이고, 유지보수성을 극대화하며, 향후 새로운 기능을 추가할 때 발생하는 '기술 부채'를 청산하기 위함.

도메인의 최소 기능 분리
각 도메인이 가장 최소한의 고유 책임과 기능만 수행하도록 격리하는 설계 원칙.

Order 도메인은 '상품 선택, 금액 계산, 주문 상태 관리'라는 본연의 역할에만 집중하고, User이나 Payment 도메인이 내부적으로 어떻게 돌아가는지는 알 필요가 없도록 만드는 것. ("너는 너, 나는 나")

인프라스트럭처 (Infrastructure)
시스템이나 소프트웨어가 정상적으로 살아 움직일 수 있도록 보이지 않는 곳에서 받쳐주는 기반 시설(서버 컴퓨터, 네트워크, 데이터베이스 등).

도메인이 최소 기능으로 분리되면, 향후 트래픽 확장에 맞춰 '주문 서버 인프라', '결제 서버 인프라' 형태로 효율적이고 유연하게 쪼개어 운영(MSA 인프라 구축)할 수 있는 토대가 됨.

2. 기존 코드의 한계점
기존 코드에서는 Order 엔티티가 User 객체와 Payment 객체를 JPA 연관관계(@ManyToOne, @OneToOne)로 직접 품고 있었습니다. 이로 인해 다음과 같은 강한 결합 문제가 발생했습니다.

성능 저하: 주문 목록을 조회할 때 원치 않는 회원 정보와 결제 정보까지 DB에서 무겁게 조인되거나 추가 쿼리(N+1 문제 등)가 발생함.

수정의 공포: 회원 도메인의 요구사항이 바뀌어 User 엔티티를 수정하면, 아무 상관 없는 Order 도메인의 비즈니스 로직이나 테스트 코드까지 줄줄이 컴파일 에러가 나거나 장애가 전파됨.

MSA 전환의 한계: 주문 시스템과 결제 시스템을 별도의 프로젝트나 서버로 독립시키고 싶어도, 코드 레벨에서 객체로 묶여 있어 찢어내는 것이 불가능함.

3. JPA & 연관관계
개념: 자바 객체와 관계형 데이터베이스(RDB) 테이블을 중간에서 매핑해 주는 ORM 기술의 표준 명세(인터페이스). (실제 구현체로 Hibernate를 주로 사용)

등장 배경: 객체지향과 관계형 DB의 관점 차이(패러다임 불일치)로 인한 노가다성 SQL 작성을 줄이고, 개발자가 객체 중심으로 비즈니스 로직에 집중할 수 있게 함.

연관관계 Association: 두 대상이 서로 연결된 상태.

객체: 필드의 '참조'로 연결 (기본적으로 단방향).

테이블: 'FK'와 JOIN으로 연결 (무조건 양방향).

@JoinColumn: 객체의 참조 변수와 테이블의 외래 키(FK) 컬럼을 매핑하는 어노테이션.

4. 도메인 주도 설계 (DDD) 핵심 개념
바운디드 컨텍스트 : 거대한 도메인을 쪼갠 "특정 용어와 비즈니스 규칙이 일관되게 적용되는 명확한 경계". (예: '상품'이 주문 부서와 배송 부서에서 각각 다른 모델로 정의됨)

애그리거트: 데이터 변경 및 일관성의 단위로 묶인 연관 객체들의 집합.

애그리거트 루트: 애그리거트의 대장 객체. 외부의 모든 요청은 오직 루트를 통해서만 이루어져야 내부 데이터 정합성이 깨지지 않음.

불변식: 어떤 상황에서도 항상 참이어야만 하는 비즈니스 제약 규칙. 애그리거트 루트가 목숨 걸고 지켜야 하는 조건 (예: 잔액은 0원 미만이 될 수 없다).

Cascade와 애그리거트: 영속성 전파(Cascade)는 데이터의 생사고락을 함께하는 동일한 애그리거트 내부에서만 사용해야 함. 다른 애그리거트끼리 거는 것은 결합도를 높여 트랜잭션 충돌 및 성능 저하를 유발하므로 금지하며, 다른 애그리거트는 ID를 통해 간접 참조하는 것이 올바름.
