MSA
1. 모놀리식은 무죄(거대한 하나의 돌덩어리)
모든 로직이 DB 하나만 바라봄

2. 장단점
장점
	1. 개발이 빠름
	2. 디버깅이 쉬움
	3. 데이터 정합성이 공짜
	4. 배포가 단순
단점
	1. 배포의 공포(오타만 고쳐도 앱을 전체 다시 띄워야함)
	2. 장애의 전염(삼품 조회에 요청 많아지면 스레드 풀 고갈)
	3. 스케일링 낭비(클라우드 시대에 서버 비용은 곧 경쟁력)
	4. 팀의 충돌(PR올릴 때마다 머지 충돌)
    
3. 그래서 쪼갠다 -> MSA
MSA: 독립적으로 배포 가능한 작은 서비스들로 나누고 API로 협력하는 구조

APi gateway는 왜 있나(아파트 경비실이라고 생각)
모든 요청이 통과하는 단일 입구
클라이언트는 게이트웨이주소 하나만 알면 됨
인증/인가를 입구에서 한번에 처리

서비스끼리는 어떻게 대화하나
동기통신 - 전화
	상대 서비스의 API를 호출하고 응답을 기다림 -REST가 대표적
비동기 통신 -문자 메시지
	메시지 큐(kafka)에 이벤트를 던져두고 할일을 계속 함

서비스별 독립DB
MSA의 핵심은 코드를 나누는 게 아니라 데이터를 나누는 것
서비스를 나눴는데 DB를 공유하면 겉만 MSA임

실제로 모놀리식으로 시작해서 MSA로 점차 바꾸지 처음부터 MSA로 시작은 거의 없음

4. MSA의 tradeoff
	비용1. 메서드 호출이 네트워크 호출이 된다 
	-"네트워크를 신뢰"할 수 있다는 착각이 있음
	비용2. @Transactional의 이점이 사라진다
	-자동 롤백은 없음, 직접 되돌려야함 => Saga패턴
	비용3. 디버깅이 어려워짐
	-요청이 게이트웨이에 들어온느 순간 고유한 추적 ID를 발급하고분산추적으로 중랄 로그 수집 시스템이 필요함
	비용4. 인프라 부담의 폭증
	-API gateway, 메세지 큐(kafka), 서킷 브레이커, 분산 추적, 중앙 로그 수집 등등을 알아야함

5. 어떻게 전환하나
빅뱅식 전환: 서비스 멈추고 전부 새로 만들어서 한 방에 교체 -> 금기
MSA 전환은 이벤트가 아니라 여정이다 -> 한 조각씩 떼어내기

애그리거트의 본질은 "비즈니스 규칙(Invariant, 불변식)을 함께 유지해야 하는 객체들의 묶음
즉 애그리거트는 트랜잭션의 단위이다
=> 1트랜잭션 = 1애그리거트
여러 애그리거트의 상태를 함께 변경해야 할 때는 하나의 큰 트랜잭션으로 묶지 말고, 도메인 이벤트를 활용해 트랜잭션을 분리하는 것이 좋다
